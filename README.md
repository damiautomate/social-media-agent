# Social Media Agent — Phase 1

Multi-user content automation for LinkedIn, Instagram, TikTok, and Facebook.
Each user signs up with Firebase Auth, brings their own Anthropic API key, and
gets an isolated brand config + drafts. Drafts are generated by a Firebase
Cloud Function (Gen 2) triggered on Firestore.

Phase 1 covers: auth, onboarding, brand config, draft generation, review
dashboard. Research agent, scheduling, publishing, and analytics come later.

## Stack

- Next.js 15 (App Router, JS) on Vercel
- Firebase Auth (email/password + Google)
- Firestore (project: `social-agent`)
- Cloud Functions Gen 2 (Node 20) for AI work
- `@anthropic-ai/sdk`, model `claude-sonnet-4-20250514`

## Local development

```
npm install
cd functions && npm install && cd ..
cp .env.example .env.local   # fill in values
npm run dev
```

The Next.js app needs both `NEXT_PUBLIC_FIREBASE_*` (client) and
`FIREBASE_SERVICE_ACCOUNT_JSON` (server). The latter is the full JSON of a
Firebase service account key, pasted as a single value.

## Firebase setup

1. Create a Firebase project named `social-agent` (or update `.firebaserc`).
2. Enable Authentication → Email/Password and Google providers.
3. Enable Firestore in production mode.
4. Generate a service account: Project settings → Service accounts → Generate
   new private key. Use this JSON for `FIREBASE_SERVICE_ACCOUNT_JSON`.
5. Get a web app config: Project settings → Your apps → Web. Use this for the
   `NEXT_PUBLIC_FIREBASE_*` env vars.

## Deploy Firestore rules + indexes + Cloud Function

```
firebase login
firebase use social-agent
firebase deploy --only firestore:rules,firestore:indexes,functions
```

The Cloud Function is `processPendingJob` (Firestore trigger on
`pending_jobs/{jobId}`). It runs as the project default service account, which
has full Firestore access — that is intentional, all client access to
`drafts` / `ideas` / `pending_jobs` is locked down by `firestore.rules`.

## Deploy Next.js to Vercel

1. Push the repo to GitHub.
2. Import to Vercel.
3. Add the environment variables from `.env.example`.
4. Deploy. First sign-up will hit `/api/auth/bootstrap`, which creates the
   `users/{uid}` doc and seeds `users/{uid}/brandConfig/main` from the default
   template.

## Multi-user model

- Every Firestore document carries a `userId` field.
- Security rules let a user read only their own `users/{uid}` doc, their
  `brandConfig` subdoc, and their own `drafts` / `ideas` (read only — writes
  go through API routes / Cloud Function via Admin SDK).
- `pending_jobs` and `analytics` are server-only end to end.
- Each user supplies their own Anthropic API key. Generations bill to that
  key. Keys are stored as plaintext on `users/{uid}.anthropicApiKey` — fine
  for the trusted small group; encrypt later if/when needed.

## Async generation flow

1. Dashboard `POST /api/generate` (with `Authorization: Bearer <id-token>`).
2. The route verifies the token, loads the user's brand config + key, and
   creates a `pending_jobs/{jobId}` document.
3. The Cloud Function picks up the new doc, builds the prompt, calls Claude
   with the user's key, writes a `drafts/{draftId}` doc, and marks the job
   `completed`.
4. The dashboard's `onSnapshot` on `drafts where userId == uid` shows the
   new draft as soon as it's written.
